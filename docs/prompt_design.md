# Prompt 设计说明

这个项目的 Prompt 目标不是一次性生成完整长篇，而是支持长期、分阶段地推进小说项目：先定总纲，再定人物，再逐章生成正文，并用摘要压缩历史上下文。

## system prompt 的作用

全局 system prompt 定义模型的写作身份和基本约束：

- 它是一名专业小说创作助手。
- 必须遵守用户给定的题材、人物、世界观和叙事风格。
- 必须保持人物性格、世界观规则和叙事节奏一致。
- 不自称 AI。
- 不解释创作过程。
- 默认直接输出作品内容。

这样做的目的，是减少模型在正文中插入“下面是我的创作思路”之类的无关说明。

## 用户设定如何结构化

页面中的输入会被整理成统一的项目设定块，包括：

- 小说标题
- 小说类型
- 写作风格
- 主角设定
- 重要配角设定
- 世界观设定
- 故事核心冲突
- 目标读者
- 单章字数范围
- 额外要求

这些字段会被转换成清晰的 Markdown 列表，放入每一种生成模式的 user prompt 中。这样比把所有内容拼成一段自然语言更稳定，也更方便后续追加字段。

## 白话设定扩写 Prompt

白话设定扩写用于把用户随手写下的故事想法，转换成页面已有的四个核心字段：主角设定、重要配角设定、世界观设定和故事核心冲突。

这个 Prompt 要求模型输出严格 JSON，原因是页面需要稳定解析并自动填入 `st.session_state`。如果模型输出 Markdown、解释文字或嵌套结构，页面就需要额外猜测格式，失败概率会变高。

字段值使用字符串，而不是复杂对象或数组，是为了直接填入 Streamlit 的多行文本框。每个字段内部仍然可以使用换行组织信息，但外层结构保持简单：

```json
{
  "title_candidates": ["..."],
  "recommended_title": "...",
  "protagonist_setting": "...",
  "supporting_characters_setting": "...",
  "world_setting": "...",
  "core_conflict": "..."
}
```

Prompt 会明确要求保留用户原始设定的核心，不随意改变题材、主角目标和核心矛盾。模型可以补充姓名、过往经历、世界规则或阻力来源，但不能覆盖用户已经明确给出的内容。

`title_candidates` 用于在用户没有明确标题时提供多个可选方向，例如短标题、网文风标题、文艺风标题、类型感强的标题和悬念感标题。`recommended_title` 必须从候选标题中选择，这样页面自动填入标题时有明确来源，也方便用户回看和替换。

如果用户已经在页面中填写了标题，扩写结果不会自动覆盖它，只会展示候选标题供参考。这样可以保护用户的明确命名意图，同时让标题候选继续作为灵感材料。标题生成也服务于 outputs 多小说隔离目录：当前标题会决定项目保存到哪个 `outputs/{标题}/` 子目录。

为了避免标题过于通用或不可用，Prompt 会要求标题结合故事核心卖点，不使用“未命名小说”“我的小说”等占位标题，并控制普通标题长度。保存前还会对标题做 Windows 路径安全化处理。

`detail_level` 控制扩写密度：

- 低：每个字段约 150-250 字。
- 中：每个字段约 300-500 字。
- 高：每个字段约 600-900 字。

三个开关用于控制模型主动补充的范围：

- 不补充配角时，只输出 1-2 个最必要配角。
- 不补充核心冲突时，只整理用户已给出的冲突，不主动增加复杂反转。
- 不补充世界规则时，只整理背景，不主动增加复杂规则。

JSON 解析时，页面会先直接 `json.loads`。如果失败，会尝试截取返回内容中第一个 `{` 到最后一个 `}` 再解析。仍然失败时，页面会显示错误原因和模型原始返回内容，方便继续调 Prompt。

## 多小说项目隔离策略

把所有输出都放在 `outputs` 根目录会让不同小说的大纲、人物卡、章节正文、摘要和配置混在一起。一键续写时尤其危险，因为系统可能误读另一部小说的上一章或摘要，造成上下文污染。

项目现在使用小说标题作为 `outputs` 下的项目目录名，例如 `outputs/赛博记忆/`。这样用户在页面切换标题时，文件读写范围也会随之切换。

大纲、人物卡、项目配置、章节索引和设定扩写结果放在项目目录根部；章节正文放入 `chapters/`；章节摘要放入 `summaries/`。这种分层让长期连载文件更容易扫描，也让“正文”和“压缩记忆”分开管理。

续写时，程序只读取当前小说项目下的 `chapters/`、`summaries/`、`novel_outline.md` 和 `characters.md`。其他小说目录不会参与 Prompt 构造，因此不会把不同故事的人物、世界观或章节摘要混入上下文。

因为小说标题会变成 Windows 文件夹名，保存前必须做安全化处理。非法路径字符会替换为 `_`，空标题会使用“未命名小说”，并限制目录名长度，避免路径错误。

## 如何控制小说风格

风格控制主要来自三个位置：

1. 页面中的“写作风格”字段，例如热血、冷峻、轻松、黑暗、文艺、网文爽文、日式轻小说、细腻现实主义。
2. “目标读者”字段，用来影响节奏、信息密度和表达方式。
3. “额外要求”字段，用来写更细的控制项，例如第一人称、第三人称、多对话、多描写、节奏快、节奏慢、禁止出现的内容或希望出现的桥段。

章节正文 Prompt 会要求模型不要输出提纲，不要解释创作思路，要写成具体场景。这能减少“剧情总结感”，让输出更接近可阅读正文。

## 如何保持人物一致性

人物一致性来自三层上下文：

1. 当前页面输入的主角和配角设定。
2. 当前小说项目中已生成的人物卡 `outputs/小说标题/characters.md`。
3. 历史章节摘要中记录的关系变化和悬念。

生成章节时，如果存在人物卡，系统会自动读取最近版本的人物卡并放入 Prompt。章节正文 Prompt 也明确要求保持人物性格一致、对话自然、不要突然引入没有铺垫的设定。

## 如何处理续写

当用户勾选“使用上一章上下文”时，项目会尝试读取上一章正文。例如生成第 3 章时，会读取最近版本的 `chapter_002*.md`。

如果上一章正文存在，它会被加入章节 Prompt，并且只保留靠近结尾的部分。这样做是为了让新章节自然承接上一章结尾，同时避免上下文无限增长。

如果上一章正文不存在，页面会给出提示，但不会阻止生成。模型仍然可以使用当前设定、大纲、人物卡和历史摘要继续创作。

## 如何减少跑题

项目通过以下方式减少跑题：

- 大纲 Prompt 固定输出结构，要求包含世界观、主线冲突、主要人物、三幕式结构、分卷规划和前 10 章大纲。
- 人物卡 Prompt 固定字段，要求写清楚欲望、恐惧、关系、成长弧线和隐藏秘密。
- 章节正文 Prompt 明确禁止输出提纲、项目符号和创作解释。
- 章节正文 Prompt 要求遵守世界观规则、人物性格和上一章结尾。
- 用户可以在“额外要求”中加入禁止内容或必须出现的桥段。

后续如果要进一步增强稳定性，可以增加“世界观规则表”“禁用设定表”和“角色口吻示例”。

## 如何使用章节摘要压缩上下文

长篇连载不能把所有历史章节全文都塞进 Prompt，否则上下文会越来越长，生成速度变慢，也更容易超出模型限制。

本项目使用以下上下文优先级：

1. 当前页面输入或 `project_config.json` 中的小说总设定。
2. 当前小说项目的人物卡 `characters.md`，如果存在。
3. 当前小说项目的小说大纲 `novel_outline.md`，如果存在。
4. 最近一章全文，只在用户勾选“使用上一章上下文”时读取。
5. 当前小说项目 `summaries/` 中的历史章节摘要。

章节生成完成后，项目会再次调用模型，为该章节生成 100 字以内摘要。摘要会保存到当前小说项目的 `summaries/`，同时写入当前项目的 `chapter_index.md`。

摘要的作用是保留长期记忆，例如：

- 本章发生了什么。
- 角色关系有什么变化。
- 遗留了什么悬念。

这样后续章节可以通过摘要了解历史进展，而不必读取所有章节全文。

## 上下文长度控制

`config.py` 中提供了几个基础上下文限制：

- `MAX_REFERENCE_CHARS`：控制大纲和人物卡的最大注入长度。
- `MAX_PREVIOUS_CHAPTER_CHARS`：控制上一章正文的最大注入长度，并优先保留结尾。
- `MAX_SUMMARIES_CHARS`：控制历史摘要的最大注入长度。

这些值不是复杂的 token 计算，只是轻量的字符级保护，符合个人本地工具的定位。后续如果需要更精确，可以加入 token 估算逻辑。
